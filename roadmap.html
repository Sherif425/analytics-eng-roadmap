How the stack fits

Snowflake (warehouse) + dbt (transformation) is the core, not the whole job. The tools are the mechanism; dimensional modeling is the judgment — deciding what tables to build, at what grain. Around the core sit ingestion, orchestration, BI and Git/CI, each of which has a managed cloud service. This roadmap teaches the core deeply, modeling as the backbone, and the rest to working literacy.

Data warehouse design & dimensional modeling

The core judgment skill. Learn it in parallel with Phase 1; apply it in your Phase 3 portfolio. Your dbt marts layer is dimensional modeling.

Grain  ★ most important
What one row represents. Declare it first (“1 row = 1 transaction”). Every other decision follows from it.
Fact vs dimension tables
Facts = measurable events (a transaction, with amount/fee). Dimensions = descriptive context (customer, merchant, date).
Star schema
One fact surrounded by dimensions joined on keys. The standard analytical shape — and exactly what your dbt marts produce.
Surrogate keys
Stable, warehouse-generated dimension keys — don't depend on fragile source-system IDs.
Slowly Changing Dimensions
How you track history when an attribute changes. Type 2 (keep old + new rows with validity dates) is the one to know cold.
Conformed dimensions
Shared dimensions used identically across multiple facts/marts — the “bus” that keeps definitions consistent.
Kimball's 4-step design
1) pick the business process, 2) declare the grain, 3) choose dimensions, 4) choose facts. Your design checklist.
dbt layering
staging → intermediate → marts. The marts layer is where your dimensional models live.
Wide table vs star (modern)
On columnar cloud warehouses a “one big table” sometimes beats a strict star. Learn the tradeoff after Kimball, not instead of it.
Apply it (payments): build fact_transactions (grain: 1 transaction) joined to dim_customer, dim_merchant, dim_date, dim_payment_method. In your case study, state the grain and justify the design — that paragraph is what makes a client trust you.

Deploying the stack on AWS

Each pipeline stage maps to a managed AWS service. Snowflake already runs on AWS, and dbt is warehouse-agnostic (Snowflake or Redshift), so most of your skills transfer directly.

Pipeline stageWhat it doesAWS-native service(s)Keep-your-stack alternative
Extract & IngestPull from sources (DBs, SaaS, streams) and land raw dataAWS Glue · AWS DMS (CDC) · Kinesis / Firehose · AppFlowFivetran or Airbyte on AWS
Load / Data lakeRaw + staged storage; the landing zoneAmazon S3 + Glue Data CatalogS3 external stages, even with Snowflake
Data warehouseStore + run analytical SQL (the compute)Amazon Redshift (or Athena over S3 / Iceberg)Snowflake-on-AWS — same dbt models run
Transform (the ‘T’)Model, test, document in the warehousedbt (Core or Cloud) on Redshift; Glue for Sparkdbt — your transferable skill, unchanged
OrchestrateSchedule, run, monitor, recoverAmazon MWAA (Managed Airflow) · Step Functionsdbt Cloud scheduler; or ECS Fargate
BI & dashboardsServe insights to business usersAmazon QuickSight (native to Redshift)Power BI or Looker connect fine too
Govern & catalogPermissions, metadata, lineage (cross-cutting)AWS Lake Formation · IAM · Glue Data Catalogsame regardless of warehouse
CI/CD & IaCReproducible, automated deploysGitHub Actions / CodePipeline · Terraform / CloudFormationengineering discipline clients value
How dbt itself gets deployed on AWS
  • dbt Cloud (managed SaaS) — simplest: point it at Redshift or Snowflake; also on AWS Marketplace.
  • Amazon MWAA (Managed Airflow) running dbt via the Cosmos library — AWS's documented pattern.
  • Containerized: an ECS Fargate scheduled task, or EKS, running dbt build.
  • Serverless: AWS Step Functions + Lambda / Fargate.
  • CI: GitHub Actions runs dbt on every merge. (Compute always happens in the warehouse.)
Two routes — pick by goal
Route A — Snowflake on AWS

Low friction, fastest freelance value. Keep Snowflake + dbt, host Snowflake in an AWS region, add S3 + IAM + QuickSight literacy. You can truthfully say “I deploy on AWS” without a big detour.

Route B — Full AWS-native

Higher ceiling. Redshift + Glue/DMS/Kinesis + dbt + MWAA + QuickSight + Lake Formation/IAM. Opens AWS-shop clients; target DEA-C01. Bigger investment — do it after you have paying work.

AWS learning sub-phases
  • AWS-1 Foundations: account, IAM, S3, VPC basics, cost/billing.
  • AWS-2 Data services: S3, Glue, Redshift, dbt-on-Redshift, MWAA, QuickSight, Kinesis/DMS — you already know what each stage does.
  • AWS-3 Deploy dbt: run on Redshift; schedule via MWAA or Fargate; GitHub Actions CI; a taste of Terraform/CloudFormation.
  • AWS-4 Portfolio + cert: rebuild the payments project AWS-native, then pursue DEA-C01.

Positioning & soft skills

Your headline
Analytics Engineer | Snowflake + dbt | dimensional modeling | Payments & Billing data. Sell judgment (how you model data), not just tool knowledge.
Why modeling is your edge
Designing clean star schemas (right grain, conformed dimensions) is what separates an analytics engineer from a dbt operator — and it's what clients rehire.
Why payments is your edge
E-payment / bill-payment data needs exactly the analytics clients pay for: volume & trends, success/decline, settlement & reconciliation, fees/revenue, cohorts, churn, fraud, compliance.
Soft skills that earn the premium
Requirements elicitation; metric & grain definition; two-way translation (tech ↔ business); scope management; clear written communication (wins Upwork bids).
